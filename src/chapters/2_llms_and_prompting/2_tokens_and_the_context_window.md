A token is a **chunk of text the model processes as a single unit.** Not a word, not a character, somewhere in between. Common English words are usually one token. Less common words split into pieces. Punctuation and whitespace tokenize their own way. Code is denser than prose.

A concrete example, using Claude's tokenizer (each `|` is a token boundary):

```
|The| capital| of| France| is| Paris|.|

|def| greet|(|name|):|
|    |return| f|"|Hello|,| {|name|}|!|"|
```

Workable rule of thumb for English prose: **1 token per ~0.75 words**, or about **4 characters per token**. Code is denser (more punctuation tokens per line). Non-English text is denser again. You can play with Anthropic's tokenizer interactively to build intuition.

Tokens matter for two reasons: they're what you pay in, and they're what fills the context window.

## The unit of cost

Every message sends two counters ticking:

- **Input tokens**: everything the model reads. Your prompt, the full conversation so far, any file contents pulled in, the system prompt on top. All of it.
- **Output tokens**: everything the model writes back.

APIs charge per million tokens, with output usually several times more expensive than input. For this course, NSS is paying through your team seat, so you're not watching a dollar counter. Good. But the habits that control token usage also control **latency** (fewer input tokens means faster responses) and **quality** (a focused context beats a sprawling one). Build the habits now, while someone else is paying.

## The context window

Every model has a **context window**: the maximum number of tokens it can hold in a single conversation. Current Claude models carry a context window in the hundreds of thousands of tokens (a million on some tiers), which sounds like a lot. It's a lot for a single prompt. It's not a lot for a long working session with several files open.

Some points of reference:

- A typical chat message: a few hundred tokens.
- A medium source file (a couple hundred lines): a couple thousand.
- A long README: a few thousand.
- A sprawling debug session with five files open and twenty turns: tens of thousands.

When the window fills, older turns get dropped or summarized. You'll learn specific tools for managing that (`/clear`, `/compact`, fresh sessions) in a later module. For now, just know the window has an edge.

## Why long conversations cost more than you'd expect

Here's the surprising result, and it's the most important part of this chapter.

An LLM has no memory of its own. Between turns, it remembers exactly nothing. The illusion of continuity is produced by the tool re-sending the whole conversation every turn. On turn two, Claude Code sends the system prompt + your first message + the model's first reply + your new message. On turn ten, everything from turns one through nine is included, plus your new message.

**Every turn re-reads the entire prior conversation as input.**

If an average turn adds about `k` tokens of new material, then per-turn input costs grow linearly: `k`, `2k`, `3k`, ..., `N·k`. But the *cumulative* cost across the whole conversation is the sum:

```
k + 2k + 3k + ... + N·k  =  k · N(N+1) / 2  ≈  k · N² / 2
```

That's quadratic. Double the conversation length, pay roughly four times as much, not twice. Put numbers on it with `k = 1,000` tokens per turn:

| Conversation length | Cumulative input tokens |
| ---: | ---: |
| 5 turns  | 15,000 |
| 10 turns | 55,000 |
| 20 turns | 210,000 |
| 30 turns | 465,000 |

**A 20-turn conversation costs roughly 4× a 10-turn one, not 2×.** A 30-turn conversation costs over 8×. Sit with that for a second. It's counterintuitive the first time you see it.

Three practical consequences, whether or not you're paying the bill:

1. **Latency.** The model reads every input token before generating output. Long sessions feel visibly sluggish.
2. **Quality degradation.** With a sprawling context, relevant details get diluted by irrelevant ones from ten turns ago. Answers get vaguer, more generic, sometimes wrong in ways that would never have happened in a fresh session.
3. **Window limits.** Eventually you bump the ceiling. Things get summarized (losing detail) or refuse to continue.

The point isn't the math above, which may or may not be correct considering this content was generated by an LLM, the point is: **short focused conversations beat long sprawling ones.** That's not aesthetic; it's the direct consequence of the quadratic curve. When a task is done, start a fresh conversation for the next task. Don't ride one session from 9am to 5pm. A later module covers the specific tools for this; for now the habit is what matters.
